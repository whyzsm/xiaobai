## 1. 被评审对象

- 项目：`xbaiProjectCode`，即 "Loop Engineering" 工程骨架。
- 定位：把 agent 工作流拆成**可维护、可审计、可验证**的系统能力；引擎层 `loop-engineering/` 与运行空间 `workspace/` 分离。

## 3. 各专家独立意见

### 3.1 二号专家 · Agent 工作流评审（Hy3）

**结论：通过项 + 3 项风险 + 1 项缺失**

- ✅ **生成/评审分离设计正确**：`workspace/agents/generator.agent.yaml` 与 `evaluator.agent.yaml` 为独立 agent；`morning-triage.loop.yaml` 明确 `verification.allowSelfReview: false`，禁止 generator 自评——这是本项目最值得保留的硬约束。
- ✅ **Evaluator 立场合理**：`evaluator.agent.yaml` 设定 `stance: skeptical`、`instructions` 要求"默认假设改动是坏的、直接看 diff、跑测试、对照项目 SKILL.md、证据不足即 reject"，符合独立评审者的应有姿态。
- ✅ **Harness 工具白名单存在**：`coding.harness.yaml` 声明允许的工具范围，方向正确。
- ⚠️ **风险 A（权限未真正生效）**：当前 harness 的工具白名单是声明式，`harness-runtime` 在骨架期并不真正绑定/拦截工具；接入真实执行前这是"纸面门禁"。
- ⚠️ **风险 B（requiredChecks 有未定义执行者）**：`verification.requiredChecks` 含 `lint / unit-tests / project-skill-review`，但 `project-skill-review` 目前没有定义由谁、如何执行，存在"列出但无落地"的缺口。
- ⚠️ **风险 C（模型路由是占位符）**：`evaluator.agent.yaml` 的 `model.preference: fast-independent-model` 是占位描述，不是真实模型路由；整套系统对"用哪个模型跑 generator / evaluator"没有可执行的选型逻辑。
- ❌ **缺失（Orchestrator 规范）**：`xiaobai.orchestrator.agent.yaml` 存在且 `role: orchestrator`，但其职责、工具、与 generator/evaluator 的衔接规范未在 agent 定义中充分展开，仅由 `loop-runtime` 做路由装配。

### 3.2 三号专家 · 系统架构评审（GLM-5.2）

**结论：亮点突出，但有"双规格漂移"与"过度拆分"两类隐患**

- ✅ **分层稳健**：引擎层（`loop-engineering/`，含 runtime、schema、CLI、模板、测试）与运行空间（`workspace/`，含 loop/agent/connector/memory/budget）边界清晰，且 `AGENTS.md` 用提交边界（忽略 `.local/`、`local.paths.yaml`）强化了该边界——这是成熟的做法。
- ✅ **细粒度职责划分**：20+ 个 runtime 包（loop-runtime、scheduler、harness-runtime、context-engine、skill-runtime、worktree-manager、connector-runtime、agent-runtime、evaluator-runtime、memory-* 系列、budget-guard、human-gate 等）基本符合单一职责，利于独立演进。
- ✅ **严格可审计**：`loop.schema.json` 对 loop/agent/budget/humanGate 等使用 `additionalProperties: false`，杜绝未知字段，提升可审计性。
- ⚠️ **隐患 A（双规格漂移）**：`morning-triage.loop.yaml` 用扁平结构（无 `workflow.stages`），而 `frontend-delivery.loop.yaml` 用 V2 的 `workflow.stages[]`（九阶段）。schema 把 `workflow` 设为可选，于是两种"方言"长期共存；且默认 `npm run dry-run`/`simulate` 只跑 `morning-triage`，V2 富路径被实战打磨得更少。
- ⚠️ **隐患 B（过度拆分 vs 自身原则）**：项目根 `SKILL.md` 第一条原则就是"简洁优先 / 不投机式扩展"，但引擎已拆出 memory-capture、memory-promote、simulation-runtime 等**仅在 mock 数据上运行**的包。在尚无真实 loop 跑通前，这部分有 YAGNI 风险，建议等真实数据流入后再按需拆分。
- 💡 **改进**：优先把 `morning-triage` 也迁移到 V2 `workflow.stages` 单一方言；connector-runtime 是未来最大风险缝，建议现在就定义"connector 错误契约"（重试/超时/部分失败/限流），而不是等真实 connector 接入时再补。

### 3.3 四号专家 · 独立批判评审（Kimi-K2.7-Code）

**结论：同意基础扎实，但必须戳破"伪系统"错觉**

- 🔴 **伪系统识别**：项目当前本质是一个**配置校验器 + dry-run 计划生成器**，不是"能跑的系统"。所有"发现→交付→验证→持久化→调度"都停在 `status: planned`。README 也承认不真实调用 LLM/不真实建 PR。风险评估：骨架给人的"完备感"可能误导决策者以为工程已基本可用——这是最大的隐性风险。
- 🔴 **未验证承诺**：声称的安全机制（human gate、budget guard、connector 写权限默认拒绝）在骨架期**未被真实故障场景验证**。例如 budget 超限、human gate 绕过、connector 写操作真实发生时的拦截路径均未经实战。
- 🟠 **文档漂移（可证伪）**：`README.md` 的"目录结构"把 `workspace/agents/` 只列了 3 个文件（coding.harness.yaml / generator.agent.yaml / evaluator.agent.yaml），而实际目录有 **8 个**（另含 frontend-generator/frontend-evaluator、suagent、xiaobai.orchestrator、frontend-delivery.harness）。`AGENTS.md` 的项目结构快照也偏旧。验证方式：直接 `ls workspace/agents/` 即可复现。**结论：文档与代码已漂移。**
- 🟠 **过度设计质疑**：同 3 号——memory-capture/pattern-promotion 等"知识沉淀晋升"能力，在一个从未真实运行的系统里属于超前抽象；建议先用最简实现跑通一个真实 loop，再决定是否保留。
- ✅ **此处认同**：生成/评审分离 + `allowSelfReview:false`、严格 schema、`engine/workspace` 分离、双语文档——这些在当前阶段就是对的，不应被动摇。

## 4. 分歧与讨论（1 号总调度组织）

**分歧点 1：memory-* 系列是否过度设计？**
- 3 号：倾向"保留但暂缓扩展"，承认有 YAGNI 信号。
- 4 号：更尖锐，主张"先跑通一个真实 loop 再决定是否保留 memory-capture/promote"。
- 2 号：不直接相关（非 agent 工作流范畴）。
- **协调结论**：采纳 4 号更激进的判据——在"首个真实 loop 跑通"之前，不新增 memory 子能力；现有 memory-* 包若仅服务 mock，标记为"待真实数据验证"。优先级 P2。

**分歧点 2：是否立即统一 loop 规格方言？**
- 3 号：强烈建议统一到 V2 `workflow.stages`。
- 4 号：同意，但补充"统一前先确认 morning-triage 的扁平结构是否仍有遗留用途"。
- **协调结论**：统一到 V2 单一方言（P1）。`morning-triage` 要么迁移到 V2，要么显式标注为 legacy 示例。

**分歧点 3：模型路由占位符要不要现在解决？**
- 2 号：认为"占位符不阻塞骨架，但必须在接真实执行前定义清楚"。
- 4 号：要求现在就把"fast-independent-model"映射成真实可选项，否则 evaluator 的独立性无保障。
- **协调结论**：在接真实执行前（P1）必须定义 generator/evaluator/orchestrator 的真实模型选型与 fallback。

## 5. 综合结论与行动清单（1 号总调度收敛）

### 5.1 总体判定

**基础扎实、纪律性超出同阶段骨架；但"它是计划器而非系统"与"规格漂移"是当前最该正视的两类风险。** 项目在可审计性、安全默认值、边界清晰度上做得明显优于一般脚手架，真正的工程难度（真实 agent 执行、LLM 评审质量、connector 故障处理）被合理延后，但必须在文档与决策层面**如实标注**这一边界，避免误导。

### 5.3 共识（四位专家一致认同）

1. 生成/评审分离 + `allowSelfReview:false` 是必须保留的硬约束。
2. `engine/workspace` 分层与严格 schema（`additionalProperties:false`）方向正确。
3. 双语文档与 `AGENTS.md` 提交边界是良好工程习惯。
4. 当前应被如实定位为"骨架/计划器"，而非"已可用的 agent 系统"。

### 5.4 遗留风险与待验证项

- 真实 LLM 调用、真实 worktree/PR、真实 connector 接入后的端到端行为**完全未验证**。
- human gate / budget guard / connector 写权限默认拒绝，缺乏真实故障注入测试。
- `metrics.jsonl` 为 mock 计划数据，无真实可观测性；无法度量"一个真实 loop 是否真的 work"。
- orchestrator 的多项目路由（app-a / t-max / harmony-wardrobe）仅在测试中验证，未经真实调度。
