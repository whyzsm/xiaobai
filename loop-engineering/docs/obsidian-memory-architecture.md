# Obsidian Memory Architecture

## Goal

Build Obsidian into the durable memory backend for Loop Engineering while keeping each project isolated in its own directory. Cross-project reuse happens through global indexes, search, and explicit context loading, not by mixing project memory files together.

This system must be complete enough for daily use:

- initialize project memory in an Obsidian vault;
- validate and diagnose memory health;
- index all project memory into a deterministic machine index;
- search across projects through the index;
- build bounded context bundles for loop and agent runs;
- capture project cases;
- promote reviewed cases into reusable patterns;
- keep every write previewable, auditable, and safe.

## Non-Goals

- Do not use Obsidian as a database that requires plugins.
- Do not delete or rewrite user-authored notes as part of normal commands.
- Do not require embeddings, network access, or a hosted vector store for the first complete version.
- Do not force all projects into one shared memory directory.
- Do not migrate unrelated Obsidian notes into this protocol.

## Directory Protocol

The vault root is the directory that contains `.obsidian/`. The memory system owns only the `88-学习/00-记忆索引` and `88-学习/10-项目记忆` subtrees unless a user explicitly provides another learning root.

```text
Obsidian Vault/
  88-学习/
    00-记忆索引/
      memory-index.json
      projects.md
      cases.md
      patterns.md
      tags.md

    10-项目记忆/
      <projectId>/
        index.md
        project-profile.md
        active-context.md
        decisions.md
        inbox.md
        loops/
          <loopId>/
            state.md
            inbox.md
            decisions.md
            runs.jsonl
            findings.jsonl
            metrics.jsonl
        cases/
          YYYY-MM-DD-<slug>.md
        patterns/
          <slug>.md
        reports/
```

## Memory Layers

### L0 Raw Run Logs

Files:

- `loops/<loopId>/runs.jsonl`
- `loops/<loopId>/findings.jsonl`
- `loops/<loopId>/metrics.jsonl`

Purpose:

- preserve append-only machine facts;
- power audits and later summarization;
- avoid manual edits because every line must remain valid JSON.

### L1 Working Memory

Files:

- `active-context.md`
- `inbox.md`
- `loops/<loopId>/state.md`
- `loops/<loopId>/inbox.md`

Purpose:

- describe current focus, open decisions, and near-term carry-over;
- be safe for humans to edit in Obsidian;
- be loaded before lower-priority cross-project memories.

### L2 Project Memory

Files:

- `index.md`
- `project-profile.md`
- `decisions.md`
- project-local `patterns/`
- project-local `cases/`

Purpose:

- store stable project identity, constraints, and decisions;
- provide the default project context for loop and agent runs;
- keep project-specific cases local until reviewed for broader reuse.

### L3 Cross-Project Knowledge

Files:

- `00-记忆索引/projects.md`
- `00-记忆索引/cases.md`
- `00-记忆索引/patterns.md`
- `00-记忆索引/memory-index.json`

Purpose:

- connect isolated project directories;
- make reusable cases and patterns discoverable;
- avoid duplicating full project memory across projects.

### L4 Retrieval Index

File:

- `00-记忆索引/memory-index.json`

Purpose:

- provide deterministic search and context assembly;
- expose enough metadata for filters, scoring, and audits;
- be regenerated safely from Markdown and JSONL source files.

### L5 Execution Memory

Files:

- project prompt notes;
- skill notes;
- reviewed patterns.

Purpose:

- influence how future agents work;
- remain traceable to source cases, decisions, or manual review.

## Frontmatter Protocol

Markdown notes managed by this system should use YAML frontmatter. The indexer must tolerate missing or malformed frontmatter, report warnings, and continue scanning.

Recommended fields:

```yaml
---
id:
title:
status: seed | active | reviewed | archived
type: project-memory | project-profile | active-context | decision | inbox | loop-state | case | pattern | report
project:
loop:
tags: []
domain: []
source: local
access: private
confidence: low | medium | high
created_at:
updated_at:
---
```

Case-specific fields:

```yaml
source_project:
source_loop:
source_run_id:
promote_score: 0
related_patterns: []
```

Pattern-specific fields:

```yaml
source_cases: []
evidence_count:
applicability:
counterexamples:
stop_conditions:
```

## Machine Index Schema

`memory-index.json` is a stable machine-readable index, not an opaque cache. It is regenerated by `loop memory index` and validated by `loop memory validate`.

```ts
interface MemoryIndex {
  schemaVersion: 1;
  generatedAt: string;
  vaultRoot?: string;
  learningRoot: string;
  globalIndexRoot: string;
  projectRoot: string;
  projects: ProjectEntry[];
  notes: NoteEntry[];
  cases: CaseEntry[];
  patterns: PatternEntry[];
  tags: TagEntry[];
  links: LinkEntry[];
  warnings: MemoryWarning[];
}
```

Every note entry should include:

```ts
interface NoteEntry {
  id: string;
  kind:
    | 'project-index'
    | 'project-profile'
    | 'active-context'
    | 'decision'
    | 'inbox'
    | 'loop-state'
    | 'case'
    | 'pattern'
    | 'report'
    | 'unknown';
  projectId?: string;
  loopId?: string;
  title: string;
  path: string;
  vaultRelativePath: string;
  obsidianLink: string;
  tags: string[];
  status?: string;
  type?: string;
  domain?: string[];
  source?: string;
  access?: string;
  confidence?: 'low' | 'medium' | 'high';
  summary: string;
  headings: string[];
  links: string[];
  keywords: string[];
  mtimeMs: number;
  sizeBytes: number;
  contentHash: string;
}
```

## Commands

All commands live under `loop memory`.

```bash
loop memory init
loop memory validate
loop memory doctor
loop memory index
loop memory search "query"
loop memory context --loop morning-triage
loop memory capture case
loop memory promote
loop memory report
```

Common flags:

- `--workspace <path>`: workspace root, default `workspace`.
- `--vault <path>`: explicit Obsidian vault root.
- `--project <id>`: project id.
- `--loop <id>`: loop id.
- `--json`: machine-readable output.
- `--write`: execute a write after preview.
- `--confirm`: execute a high-impact write after preview.

Write commands must default to preview mode. Preview output must include planned file paths, created files, skipped existing files, warnings, and required confirmation flags.

### Command Examples

## Search Design

Search must be deterministic and offline. The first complete version uses keyword, metadata, and lightweight ranking from the generated index.

Score components:

```text
keywordMatch * 30
+ titleMatch * 20
+ tagOverlap * 20
+ sameProjectBoost * 15
+ confidenceBoost * 10
+ recencyBoost * 5
+ promoteScoreBoost
```

Supported filters:

- `--project`
- `--type`
- `--tag`
- `--confidence`
- `--limit`
- `--json`

The search command must not silently scan the vault if `memory-index.json` is missing. It should report that the user must run `loop memory index`.

## Context Loading

`loop memory context` builds a bounded context bundle for loop and agent runs.

Load order:

1. current project `project-profile.md`;
2. current project `active-context.md`;
3. current project `decisions.md`;
4. selected loop `state.md`;
5. selected loop `inbox.md`;
6. top ranked local cases;
7. top ranked local patterns;
8. top ranked cross-project cases and patterns;
9. omitted items with reasons.

Budget behavior:

- default to the harness `maxCharacters` when available;
- never break JSONL line structure;
- omit low-priority items before truncating high-priority Markdown;
- report included and omitted paths;
- include character estimates for every included item.

`LoopRuntime.dryRun` should expose enough context metadata to show that Obsidian memory participated in planning.

## Capture And Promotion

### Capture Case

`loop memory capture case` writes project-local cases.

Default target:

```text
88-学习/10-项目记忆/<projectId>/cases/YYYY-MM-DD-<slug>.md
```

Required sections:

- Trigger
- Symptom
- Rule
- Anti-Pattern
- Scope
- Evidence
- Reuse Hint

The command must avoid overwriting existing files by adding a suffix when needed.

### Promote Pattern

`loop memory promote` turns reviewed cases into project-local or global patterns.

Promotion requirements:

- explicit `--confirm`;
- at least one source case;
- generated pattern includes applicability, counterexamples, stop conditions, source evidence, and reuse hint;
- global index and `patterns.md` are updated after promotion;
- source cases keep links to promoted patterns.

## Validation And Doctor

`loop memory validate` checks protocol correctness:

- required directories;
- required files;
- schema validity;
- JSONL validity;
- duplicate ids;
- broken Obsidian links when detectable;
- index freshness by content hash.

`loop memory doctor` produces a health report:

- score;
- oversized files;
- orphan notes;
- missing tags;
- missing confidence;
- stale active context;
- cases with high promote score but no review;
- indexed files missing from disk;
- disk files missing from index.

Doctor must not modify files.

## Safety

Safety defaults:

- no delete command in the first complete version;
- no cleanup or dedup command;
- no recursive write outside the configured memory roots;
- no following symlinks during scanning;
- atomic index writes using a temporary file and rename;
- preview-first for `init`, `index`, `capture`, `promote`, and `report`;
- `--confirm` required for promotion and any operation that updates existing index Markdown.

## Recovery

The system treats Markdown and JSONL files as source of truth. `memory-index.json` is recoverable and can be regenerated.

Recovery steps:

1. Run `loop memory validate --json` to identify missing required files, invalid JSONL, duplicate ids, or stale index entries.
2. If only `memory-index.json` is missing or stale, run `loop memory index --write --json`.
3. If an index write is interrupted, remove only `memory-index.json.tmp` after inspecting it, then rerun `loop memory index --write --json`.
4. If a generated case or pattern is incorrect, manually archive it by changing `status` to `archived`; do not delete it during automated recovery.
5. If JSONL is malformed, fix the specific line reported by `validate`; do not reformat the entire file.
6. If Obsidian sync creates conflicts, keep both conflict files, run `doctor`, then merge manually before regenerating the index.

No recovery command should delete notes automatically in the first complete version.

## Testing Strategy

Tests must use temporary vaults and temporary workspaces. They must not touch the user's real Obsidian vault.

Required coverage:

- path resolution with absolute and workspace-relative paths;
- frontmatter parsing, including malformed frontmatter;
- Markdown extraction for title, headings, links, summary, and tags;
- index generation across multiple projects and loops;
- schema validation;
- search scoring and filters;
- context bundle budget and omission behavior;
- preview/write behavior for init, index, capture, and promote;
- CLI JSON output;
- integration with existing dry-run and simulation flows.
